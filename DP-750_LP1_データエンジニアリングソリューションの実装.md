## モジュール 1: Spark Structured Streaming によるインクリメンタル処理

### 学習目標
- Spark Structured Streaming の仕組みを理解する
- 構造化ストリーミングを最適化する技術を習得する
- 遅延到着・順序外れのイベントを処理する方法を学ぶ
- インクリメンタル処理のためのリアルタイムデータソースを設定する

---

### 1-1. Spark Structured Streaming の基本概念

**Structured Streaming** は、Spark の DataFrame API をベースとしたスケーラブルかつフォールトトレラントなストリーム処理エンジン。バッチ処理と同じ API でストリーミングを記述できるため、コードの一貫性が保たれる。

ストリーミングクエリは「無限テーブル」に対する継続的クエリとして表現される。新しいデータが到着するたびに結果テーブルが更新され、出力先（シンク）に書き出される。

### 1-2. リアルタイムデータソースの設定

| ソース | 用途 | 特徴 |
|---|---|---|
| Apache Kafka | メッセージブローカー | 高スループット・低レイテンシ |
| Azure Event Hubs | クラウドイベント取り込み | Kafka プロトコル互換 |
| Auto Loader | クラウドストレージからのファイル取り込み | Delta テーブルへの効率的な増分ロード |
| Delta テーブル（CDF） | Delta ソースからの変更読み取り | Change Data Feed 活用 |

**Auto Loader** は Azure Blob Storage / ADLS Gen2 から新着ファイルを自動検出し、スキーマ推論・スキーマ進化に対応。大量ファイル環境でも効率的に動作する。

### 1-3. トリガーモードの選択

| トリガーモード | 動作 | ユースケース |
|---|---|---|
| `Trigger.ProcessingTime("interval")` | 指定間隔ごとに処理 | 定期的なバッチ更新 |
| `Trigger.Once()` | 利用可能なデータを一度だけ処理（非推奨） | 旧来のバッチ処理 |
| `Trigger.AvailableNow` | 現時点で利用可能な全データを処理して停止 | スケジュールベースの増分処理 |
| `Trigger.Continuous("interval")` | 超低レイテンシの連続処理（実験的） | ミリ秒単位の要件 |

**試験ポイント**: `Trigger.AvailableNow` は `Trigger.Once()` の後継。複数マイクロバッチで処理するため、`Once()` より効率的。

### 1-4. ウォーターマークと遅延イベントの処理

**ウォーターマーク**は、遅延到着データを許容する最大遅延時間を定義する仕組み。ウォーターマーク以降に到着したデータは破棄され、不要な状態の蓄積を防ぐ。

```python
# ウォーターマークの設定例
df.withWatermark("event_time", "10 minutes") \
  .groupBy(window("event_time", "5 minutes")) \
  .count()
```

複数ストリームを結合する場合、グローバルウォーターマークはデフォルトで**最小値**が使われる（データ損失防止のため）。`spark.sql.streaming.multipleWatermarkPolicy` を `max` に設定すると最大値を使用できる。

### 1-5. 出力モードの選択

| 出力モード | 説明 | 適用条件 |
|---|---|---|
| `Append` | 新しい行のみ出力 | ウィンドウ集計 + ウォーターマーク必須 |
| `Complete` | 結果テーブル全体を毎回出力 | 集計クエリ全般 |
| `Update` | 変更された行のみ出力 | 集計クエリ（シンクが更新をサポートする場合） |

## モジュール 2: Delta Live Tables によるストリーミングアーキテクチャパターン

### 学習目標
- Delta Live Tables でイベント駆動アーキテクチャを構築する
- 構造化ストリーミングでデータを取り込む
- データ整合性と信頼性を確保する
- ストリーミングワークロードをスケールさせる

---

### 2-1. Delta Live Tables（DLT）の概要

**Delta Live Tables** は、信頼性の高いデータパイプラインを宣言的に構築するためのフレームワーク。パイプラインの依存関係を自動解決し、データ品質制約の適用、エラーハンドリング、モニタリングを標準機能として提供する。

DLT パイプラインのテーブルには2種類ある：

| 種類 | 特徴 | 宣言方法（Python） |
|---|---|---|
| マテリアライズドビュー | バッチ処理向け・完全更新 | `@dlt.table` |
| ストリーミングテーブル | インクリメンタル処理向け・新着データのみ処理 | `@dlt.table` + `dlt.read_stream()` |

### 2-2. イベント駆動アーキテクチャとの統合

DLT はイベント駆動アーキテクチャの**コンシューマー側**として機能する。Kafka や Event Hubs からのメッセージを受け取り、Bronze → Silver → Gold のメダリオンアーキテクチャに沿ってデータを変換する。

```
[Kafka / Event Hubs] → [Bronze: 生データ取り込み] → [Silver: クレンジング・結合] → [Gold: 集計・BI 向け]
```

### 2-3. データ整合性と信頼性の確保

**期待値（Expectations）** により、データ品質ルールをパイプライン定義に組み込む：

| アクション | 動作 | 用途 |
|---|---|---|
| `expect` | 違反行を記録するが処理は継続 | 品質メトリクスのモニタリング |
| `expect_or_drop` | 違反行を破棄 | 不正データの排除 |
| `expect_or_fail` | 違反時にパイプラインを停止 | クリティカルな品質要件 |

### 2-4. ストリーミングワークロードのスケーリング

DLT パイプラインは**拡張オートスケーリング**（Enhanced Autoscaling）をサポートし、ストリーミング負荷に応じてクラスターノード数を動的に調整する。サーバーレスコンピューティングを選択することで、インフラ管理を Databricks に委任できる。

## モジュール 3: Spark と Delta Live Tables のパフォーマンス最適化

### 学習目標
- Delta Live Tables でサーバーレスコンピューティングと並列処理を活用する
- コストベース最適化とクエリチューニングを実施する
- Change Data Capture（CDC）を使用する
- 拡張オートスケーリング機能を適用する
- オブザーバビリティとデータ品質メトリクスを実装する

---

### 3-1. サーバーレスコンピューティングとコスト最適化

サーバーレスパイプラインでは、Databricks がコンピューティングリソースを完全に管理する。クラシックパイプラインと比較して：

| 比較項目 | クラシック（プロビジョニング済み） | サーバーレス |
|---|---|---|
| インフラ管理 | ユーザーが管理 | Databricks が自動管理 |
| 起動時間 | 数分かかる場合あり | 高速 |
| コスト予測 | 固定的 | 使用量ベース |
| 推奨用途 | 長時間稼働・特定設定が必要 | 標準的な DLT パイプライン |

### 3-2. コストベース最適化（CBO）とクエリチューニング

**コストベース最適化（Cost-Based Optimization: CBO）** は、テーブル統計情報をもとに最適な実行計画を選択する機能。`ANALYZE TABLE` コマンドで統計情報を収集する。

その他の主要な最適化手法：

| 手法 | 説明 | 効果 |
|---|---|---|
| 液体クラスタリング（Liquid Clustering） | データの物理配置を自動最適化 | 従来の ZORDER より柔軟・高速 |
| Delta キャッシュ | よく使われるデータをSSDにキャッシュ | 繰り返しクエリを高速化 |
| Predictive Optimization | 統計・最適化を自動実行 | 手動 OPTIMIZE 不要 |
| Photon | ネイティブなベクトル化実行エンジン | CPU 効率を大幅改善 |

**試験ポイント**: **液体クラスタリング**は従来の `PARTITION BY` や `ZORDER` の代替として推奨される。クラスタリング列をあとから変更できる柔軟性が特徴。

### 3-3. Change Data Capture（CDC）の実装

**CDC** はソースシステムの変更（INSERT / UPDATE / DELETE）を効率的に伝播する仕組み。DLT では `APPLY CHANGES INTO` API（AUTO CDC）を使って宣言的に実装できる。

SCD（緩やかに変化するディメンション）のサポート：

| SCD タイプ | 動作 | `APPLY CHANGES` の設定 |
|---|---|---|
| Type 1 | 最新値で上書き（履歴なし） | デフォルト動作 |
| Type 2 | 履歴を保持（`__START_AT` / `__END_AT` 列で管理） | `STORED AS SCD TYPE 2` |

順序外れのイベントは、シーケンス列（単調増加の値）を指定することで自動処理される。

### 3-4. オブザーバビリティとデータ品質メトリクス

DLT パイプラインのイベントログ（`event_log` 関数）を通じて、処理行数・品質違反数・エラー情報を記録・分析できる。これにより SLA 監視やアラート設定が可能になる。

## モジュール 4: Azure Databricks における CI/CD ワークフローの実装

### 学習目標
- バージョン管理と Git 統合を実装する
- ユニットテストと統合テストを実施する
- 環境と構成を管理する
- ロールバックとロールフォワード戦略を実装する

---

### 4-1. バージョン管理と Git 統合

Azure Databricks は **Databricks Repos**（Git フォルダー）を通じて GitHub・Azure Repos・Bitbucket などの Git プロバイダーと統合される。ノートブック・ライブラリ・ジョブ定義をコードとして管理できる。

**Databricks Asset Bundles（DAB）** は、ジョブ・パイプライン・クラスター設定などの Databricks リソースを YAML ファイルとして定義し、プロジェクト全体を単一ユニットとしてデプロイする手法（推奨アプローチ）。

```yaml
# databricks.yml の例
bundle:
  name: my-data-pipeline

targets:
  dev:
    workspace:
      host: https://<workspace-url>
  prod:
    workspace:
      host: https://<prod-workspace-url>
```

### 4-2. テスト戦略

| テスト種別 | 目的 | ツール例 |
|---|---|---|
| ユニットテスト | 個々の関数・変換ロジックの検証 | pytest、nutter |
| 統合テスト | パイプライン全体の動作検証 | Databricks テストクラスター上での実行 |
| データ品質テスト | スキーマ・値の範囲・整合性の検証 | Great Expectations、DLT Expectations |

### 4-3. 環境管理とロールバック戦略

複数環境（開発・ステージング・本番）を分離するベストプラクティス：

- 各環境に独立したワークスペースを使用する
- 環境変数やパラメーターで設定値を切り替える
- DAB の `targets` セクションで環境別設定を管理する

**ロールバック戦略**：

| 戦略 | 方法 | 適用場面 |
|---|---|---|
| ロールバック | 以前の Git コミットに戻す | コードの問題 |
| Delta タイムトラベル | `RESTORE TABLE` で過去の状態に復元 | データの問題 |
| ロールフォワード | バグ修正コミットを追加してデプロイ | 小規模な修正 |

**認証のベストプラクティス**: CI/CD パイプラインの認証には**ワークロード ID フェデレーション**（Workload Identity Federation）を推奨。シークレットを使用せずに安全に認証できる。

## モジュール 5: Azure Databricks Jobs によるワークロードの自動化

### 学習目標
- ジョブスケジューリングと自動化を実装する
- パラメーターでワークフローを最適化する
- 依存関係を管理する
- エラーハンドリングとリトライ機構を実装する

---

### 5-1. ジョブスケジューリングと自動化

**Azure Databricks Jobs** は、複数のタスクを組み合わせた複雑なワークフローをスケジュール実行するオーケストレーション機能。**Lakeflow Jobs**（旧称: Databricks Workflows）として提供される。

タスクの種類：

| タスク種別 | 説明 |
|---|---|
| ノートブック | Databricks ノートブックを実行 |
| Python スクリプト | .py ファイルを実行 |
| JAR | Java/Scala JAR ファイルを実行 |
| Delta Live Tables パイプライン | DLT パイプラインをトリガー |
| dbt | dbt プロジェクトを実行 |
| SQL | SQL ウェアハウス上でクエリを実行 |

スケジュールは **Cron 式**または UI で設定し、タイムゾーンも指定できる。

### 5-2. パラメーターと依存関係管理

**ジョブパラメーター**を使うと、同一ジョブ定義を異なる設定で再利用できる。`{{job.parameters.param_name}}` の形式でタスク間でパラメーターを参照できる。

**タスク依存関係**の設定により、DAG（有向非巡回グラフ）形式のワークフローを構築できる：

```
[データ取り込みタスク] → [変換タスク A] → [集計タスク]
                       → [変換タスク B] ↗
```

`valueFrom` を使うと、上流タスクの出力値を下流タスクのパラメーターとして渡すことができる（タスク値の伝播）。

### 5-3. エラーハンドリングとリトライ

| 設定項目 | 説明 |
|---|---|
| 最大リトライ回数 | 失敗時に再試行する回数 |
| リトライ間隔 | リトライ間の待機時間 |
| タイムアウト | タスクの最大実行時間 |
| 失敗時のアクション | 以降のタスクをスキップ / 停止 / 継続 |

アラート設定により、ジョブの開始・成功・失敗・期間超過をメールや Webhook で通知できる。

## モジュール 6: Azure Databricks によるデータプライバシーとガバナンス

### 学習目標
- データ暗号化技術を実装する
- アクセス制御を管理する
- データマスキングと匿名化を実装する
- コンプライアンスフレームワークとセキュアなデータ共有を活用する
- データリネージとメタデータ管理を使用する
- ガバナンスの自動化を展開する

---

### 6-1. Unity Catalog によるガバナンス基盤

**Unity Catalog** は Azure Databricks の統合ガバナンスソリューション。アカウントレベルで一元管理し、複数のワークスペースをまたいで権限を統制する。

オブジェクト階層：

```
アカウント
  └─ カタログ（Catalog）
       └─ スキーマ（Schema / Database）
            └─ テーブル / ビュー / ボリューム / 関数
```

権限は `GRANT` / `REVOKE` 構文で管理し、継承モデルにより上位オブジェクトの権限が下位に伝播する。

### 6-2. データ暗号化とアクセス制御

| セキュリティ機能 | 説明 |
|---|---|
| 保存データの暗号化 | Delta テーブルは Azure Storage の暗号化（AES-256）で保護 |
| 転送中の暗号化 | TLS 1.2 以上による通信暗号化 |
| カスタマーマネージドキー | Azure Key Vault でキーを管理 |
| プライベートエンドポイント | ネットワーク分離でパブリックアクセスを遮断 |

アクセス制御の粒度：カタログ → スキーマ → テーブル → 列 → 行レベルまで細かく制御できる。

### 6-3. データマスキングと匿名化

**列マスク（Column Masks）**: クエリ実行時に動的にマスキング関数を適用。データを複製せずに感度の高い列を保護する。

**行フィルター（Row Filters）**: クエリ実行時に動的にフィルター述語を適用。ユーザーのグループに応じて表示できる行を制限する。

```sql
-- 列マスクの例: 管理者以外はメールアドレスをマスク
CREATE FUNCTION mask_email(email STRING)
  RETURN CASE WHEN is_account_group_member('admins') THEN email
              ELSE CONCAT(LEFT(email, 2), '***@***.com') END;

ALTER TABLE customers ALTER COLUMN email SET MASK mask_email;
```

### 6-4. データリネージとメタデータ管理

Unity Catalog は**データリネージ**を自動的にキャプチャする：

- **対象**: クラスター・SQL ウェアハウス上で実行されるすべてのクエリ
- **粒度**: テーブルレベル・列レベルのリネージ
- **保持期間**: 1年間
- **関連情報**: ノートブック・ジョブ・ダッシュボードも含めた全体の系譜

**監査ログ**はシステムテーブル（`system.access.audit`）として提供され、誰がいつ何のデータにアクセスしたかを SQL でクエリできる。

### 6-5. デルタ共有とコンプライアンス

**Delta Sharing** は、Databricks プラットフォームを超えてデータを安全に共有するオープンプロトコル。受信者は Databricks を持っていなくてもデータを受け取れる。GDPR・HIPAA などのコンプライアンス要件に対応したデータ共有が可能。

## モジュール 7: Azure Databricks での SQL ウェアハウスの使用

### 学習目標
- Azure Databricks で SQL ウェアハウスを作成・設定する
- データベースとテーブルを作成する
- クエリとダッシュボードを作成する

---

### 7-1. SQL ウェアハウスとは

**SQL ウェアハウス**（旧称: SQL エンドポイント）は、データアナリストが馴染みのある SQL クエリでデータを操作するためのコンピューティングリソース。Databricks SQL（DBSQL）のインターフェースと組み合わせて使用する。

| ウェアハウスの種類 | 説明 | 推奨用途 |
|---|---|---|
| サーバーレス | Databricks が管理・最適化 | 標準的な BI ワークロード |
| プロビジョニング済み | ユーザーがクラスター設定を管理 | 特定の設定が必要な場合 |
| クラシック | 旧来のタイプ（非推奨） | 移行対象 |

**Auto Stop** 機能により、一定時間アイドル状態が続いたウェアハウスを自動的に停止してコストを節約できる。

### 7-2. データベース・テーブルと SQL の操作

SQL ウェアハウスでは Unity Catalog のオブジェクト階層（カタログ・スキーマ・テーブル）を標準 SQL で操作できる。**Delta テーブル**が主要なストレージ形式として使われ、ACID トランザクション・タイムトラベルをサポートする。

**クエリエディター**では：
- SQL クエリの作成・保存・共有
- 可視化（チャート・ピボットテーブル等）の追加
- クエリパラメーターによる動的フィルタリング

### 7-3. ダッシュボードとデータ共有

**Databricks AI/BI ダッシュボード**（旧称: Databricks SQL ダッシュボード）を使って複数のクエリ結果を一画面に集約し、意思決定者向けのレポートを構築できる。ダッシュボードは組織内のユーザーと共有可能。

## モジュール 8: Azure Data Factory による Databricks ノートブック実行

### 学習目標
- パイプライン内で Databricks ノートブックを実行する方法を説明する
- Azure Databricks 向けの Azure Data Factory リンクサービスを作成する
- パイプライン内で Notebook アクティビティを使用する
- ノートブックにパラメーターを渡す

---

### 8-1. ADF と Databricks の統合概要

**Azure Data Factory（ADF）** は、データ移動・変換を管理するクラウドスケールの ETL/ELT サービス。ADF パイプラインから Databricks ノートブックを呼び出すことで、ADF のオーケストレーション機能と Databricks の処理能力を組み合わせられる。

**Databricks Jobs との使い分け**：

| 観点 | Databricks Jobs | ADF + Databricks |
|---|---|---|
| 主な用途 | Databricks 中心のワークフロー | 複数の Azure サービスを統合したパイプライン |
| トリガー | スケジュール・API | スケジュール・イベント・手動 |
| 他サービス連携 | 限定的 | ADF がサポートする全コネクタ |
| 推奨シナリオ | Databricks 内で完結する処理 | 既存の ADF パイプラインへの組み込み |

### 8-2. リンクサービスの作成と認証

ADF から Databricks に接続するための**リンクサービス**を作成する際に必要な設定：

| 設定項目 | 説明 |
|---|---|
| Databricks ワークスペース URL | 接続先のワークスペースを指定 |
| 認証方式 | マネージド ID（推奨）またはアクセストークン |
| クラスター種別 | 既存クラスター / 新規ジョブクラスター（都度起動） / 新規インタラクティブクラスター |

**マネージド ID** を使った認証が推奨。サービスプリンシパルのシークレット管理が不要で、Azure のセキュリティ基盤と統合できる。

### 8-3. Notebook アクティビティとパラメーターの受け渡し

ADF パイプラインに **Notebook アクティビティ**を追加し、実行するノートブックのパスを指定する。パラメーターは次の方法で渡す：

1. **Base パラメーター**: ADF から Notebook に値を渡す（`dbutils.widgets.get("param_name")` で受け取る）
2. **ADF パイプラインパラメーター**: ADF パイプライン変数を Notebook パラメーターにマッピング
3. **出力値**: Notebook の最後のセルで `dbutils.notebook.exit("value")` を実行し、ADF が戻り値を受け取れる

モニタリングは ADF モニターと Databricks の両方で確認できる。

## まとめ: 重要ポイント

### 各モジュールのキーコンセプト

| モジュール | 試験で問われる重要ポイント |
|---|---|
| M1: Structured Streaming | `Trigger.AvailableNow` の特徴、ウォーターマークの役割、出力モードの使い分け |
| M2: DLT アーキテクチャ | マテリアライズドビュー vs ストリーミングテーブル、Expectations の3種類のアクション |
| M3: パフォーマンス最適化 | 液体クラスタリング vs ZORDER、SCD Type 1/2 の違い、CDC の `APPLY CHANGES INTO` |
| M4: CI/CD | DAB の役割、ワークロード ID フェデレーション、ロールバック vs タイムトラベル |
| M5: Jobs | タスク種別、パラメーター伝播、リトライ設定 |
| M6: ガバナンス | Unity Catalog オブジェクト階層、列マスク vs 行フィルター、データリネージの保持期間（1年） |
| M7: SQL ウェアハウス | サーバーレス vs プロビジョニング済み、Auto Stop、ダッシュボードの用途 |
| M8: ADF 連携 | マネージド ID 認証、Notebook アクティビティ、Jobs との使い分け |

### アーキテクチャ選択の判断軸

```
リアルタイム処理が必要か？
  ├─ YES → Structured Streaming / Delta Live Tables（ストリーミングテーブル）
  └─ NO  → バッチ処理（DLT マテリアライズドビュー / Spark バッチ）

データ品質制約を宣言的に管理したいか？
  ├─ YES → Delta Live Tables + Expectations
  └─ NO  → 通常の Spark DataFrame 処理

複数 Azure サービスと統合するパイプラインか？
  ├─ YES → Azure Data Factory + Databricks
  └─ NO  → Databricks Jobs（Lakeflow Jobs）のみで完結

コードのデプロイを自動化したいか？
  └─ Databricks Asset Bundles（DAB）+ GitHub Actions / Azure DevOps
```

### Unity Catalog のオブジェクト権限モデル

```
カタログ権限
  ↓（継承）
スキーマ権限
  ↓（継承）
テーブル権限 → 列マスク・行フィルターで細粒度制御
```
