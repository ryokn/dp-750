### 1-1. ワークスペースのプロビジョニング

ワークスペースは Azure Databricks の展開単位であり、Databricks アセット（ノートブック、クラスター、ジョブ、ライブラリなど）を一元管理する協調作業環境。

**作成方法**
- Azure ポータル UI
- ARM テンプレート / Bicep / Terraform
- Azure PowerShell（`New-AzDatabricksWorkspace`）
- Azure CLI（`az databricks workspace create`）

**価格レベルの比較**

| レベル | 主な特徴 |
|---|---|
| **Standard** | Apache Spark のコア機能 + Microsoft Entra ID 統合 |
| **Premium** | ロールベースのアクセス制御などエンタープライズ機能（SQL ウェアハウスはこのレベル以上が必要） |
| **Trial** | Premium レベルの 14 日間無料試用 |

> **試験ポイント**: SQL ウェアハウスは **Premium（以上）** でのみ利用可能。Standard ワークスペースでは使えない。

### 1-2. Azure Databricks のワークロード

| ワークロード | 概要 | 主なユーザー |
|---|---|---|
| **データエンジニアリング** | Apache Spark によるビッグデータ処理、ETL パイプライン構築 | データエンジニア |

### 1-3. 主要コンセプト

**クラスター**

Azure Databricks の中核となる計算エンジン。ドライバーノード（実行を調整）と1台以上のワーカーノード（分散処理を担当）で構成。

| 種類 | 特徴 |
|---|---|
| **サーバーレスコンピュート** | Databricks がフルマネージド。自動スケール、起動時間 2〜6 秒 |
| **クラシックコンピュート** | ユーザーが VM サイズ・ライブラリ・ランタイムを設定。自社 Azure サブスクリプション内で動作 |
| **SQL ウェアハウス** | SQL 分析・BI クエリに最適化。サーバーレスまたはクラシックを選択可能 |

**Delta Lake**

クラウドオブジェクトストレージ上にトランザクション機能を追加するオープンソースのストレージフレームワーク。Azure Databricks の**デフォルトテーブル形式**。

- ACID トランザクション（原子性・一貫性・分離性・耐久性）
- スケーラブルなメタデータ管理（数十億ファイル対応）
- タイムトラベル（データのバージョン管理・ロールバック）
- バッチ処理とストリーミング処理の統合

**Lakeflow Jobs**

ワークフロー自動化と実行オーケストレーション。データパイプラインの ETL 処理、ML モデルのトレーニング、ダッシュボードの更新などを自動化。スケジュール実行・イベント起動・手動実行に対応。

**Databricks Runtime**

Apache Spark のカスタマイズビルドで、パフォーマンス向上と追加ライブラリを含む。LTS（長期サポート）リリースあり。ライフサイクル: Legacy → Deprecated → End of Support → End of Life

**MLflow**

ML ライフサイクル全体を管理するオープンソースプラットフォーム。実験のトラッキング、モデル管理、開発から本番へのデプロイを支援。

### 1-4. データガバナンス（Unity Catalog と Microsoft Purview）

**Unity Catalog の 3 層名前空間**

```
catalog.schema.table_or_other_object
```

| 層 | 説明 |
|---|---|
| **カタログ（Catalog）** | チームや環境（開発・本番等）に対応するトップレベルのグループ化 |
| **スキーマ（Schema）** | カタログ内の細かな分類。プロジェクトや用途単位で整理 |
| **オブジェクト** | テーブル（マネージド/外部）、ビュー、ボリューム、関数、モデル |

**マネージドテーブル vs 外部テーブル**

| 項目 | マネージドテーブル | 外部テーブル |
|---|---|---|
| ストレージ管理 | Unity Catalog が管理 | 外部で管理 |
| データ形式 | Delta Lake 固定 | Delta, CSV, JSON, Parquet など |
| テーブル削除時 | データファイルも削除 | メタデータのみ削除、データは残存 |

**Microsoft Purview との統合**

| 機能 | 内容 |
|---|---|
| データスキャン | パブリック/プライベートネットワーク両対応 |
| メタデータ取得 | Unity Catalog のメタストア・カタログ・スキーマ・テーブル・列情報を取得 |
| 自動分類 | 組み込みルールまたはカスタムルールで機密データを識別 |
| データリネージ | データの変換・移動の追跡（Databricks 内処理を含む） |
| スキャンスケジュール | オンデマンドまたは日次/週次/月次の定期実行 |

### 2-1. アカウント階層

Azure Databricks のリソースはアカウントを頂点とした階層構造で管理される。

```
アカウント（Account）
  ├── メタストア（Metastore）  ← Unity Catalog のトップレベルメタデータコンテナ（リージョン単位）
  │     └── カタログ > スキーマ > オブジェクト（3 層名前空間）
  └── ワークスペース（Workspace）  ← 協調作業環境（複数作成可能）
        └── メタストアにアタッチ（同一リージョン内で複数ワークスペースが共有可能）
```

**アカウントで管理する内容**
- ID とアクセス（ユーザー、グループ、サービスプリンシパル）
- ワークスペースの作成・更新・削除
- Unity Catalog メタストアの管理
- 利用状況・請求・ポリシー

**Hive メタストア（レガシー）との比較**

| 項目 | Hive メタストア | Unity Catalog |
|---|---|---|
| 有効範囲 | ワークスペースごと | アカウントレベル（複数ワークスペースで共有） |
| アクセス制御定義 | ワークスペースごとに設定 | 一度設定して全ワークスペースに一貫適用 |
| 監査・ガバナンス | 限定的 | 組み込みの監査ログ、列レベルリネージ |
| 推奨状況 | レガシー（移行推奨） | 現在の推奨標準 |

> **試験ポイント**: Unity Catalog 有効化後、レガシー Hive メタストアは `hive_metastore` というカタログ名でアクセス可能になる（`hive_metastore.<schema>.<table>` の形式）。

---

### 2-2. コントロールプレーンとコンピュートプレーン

| 項目 | コントロールプレーン | コンピュートプレーン |
|---|---|---|
| 管理者 | Azure Databricks（Microsoft）が管理 | ユーザーの Azure サブスクリプションまたは Databricks アカウント |
| 役割 | Web アプリ、ジョブスケジューリング、クラスター管理（データ処理は行わない） | 実際のデータ処理 |
| 種類 | 1 種類のみ | クラシックコンピュートプレーン / サーバーレスコンピュートプレーン |

**クラシックコンピュートプレーン vs サーバーレスコンピュートプレーン**

| 項目 | クラシック（ハイブリッドワークスペース） | サーバーレス |
|---|---|---|
| 動作場所 | ユーザーの Azure サブスクリプション内の仮想ネットワーク | Databricks の Azure アカウント内 |
| 制御レベル | ネットワーク・セキュリティの詳細設定が可能 | インフラ管理不要、Databricks が自動管理 |
| 分離 | Azure サブスクリプション単位で自然な分離 | ワークスペースごとのネットワーク境界 + 追加セキュリティ制御 |

---

### 2-3. ワークスペースストレージ

**クラシックワークスペース**のストレージは Azure サブスクリプション内のストレージアカウントに格納。以下のデータを含む：

| 種類 | 内容 |
|---|---|
| ワークスペースシステムデータ | ノートブック履歴、ジョブ実行結果、クラスターログなど |
| デフォルトワークスペースカタログ | Unity Catalog 自動有効化時に作成。全ユーザーがデフォルトスキーマにアセット作成可能 |
| DBFS（レガシー） | `dbfs:/` 名前空間の分散ファイルシステム。**非推奨**。Unity Catalog のテーブル・ボリュームへの移行を推奨 |

> **注意**: ワークスペースストレージを削除・変更するとワークスペースが回復不能になる。

**サーバーレスワークスペース**は Databricks が管理する**デフォルトストレージ**を使用。ユーザーのクラウドストレージへの接続も可能。

### 3-1. Microsoft Fabric との統合

Fabric と Azure Databricks の統合は**双方向**。

**パターン 1: Fabric から Unity Catalog データを読み取る**

| 項目 | 内容 |
|---|---|
| 仕組み | Mirrored Azure Databricks Catalog を作成 → OneLake ショートカットを自動生成 |
| データ複製 | **なし**（Unity Catalog の Open API で認証情報を取得し直接アクセス） |
| 認証 | 短命の認証情報（1 時間ごとに更新、Unity Catalog からいつでも失効可能） |
| 必要権限 | 接続設定ユーザーが対象スキーマに `EXTERNAL USE SCHEMA` 権限を持つ必要あり |

**重要なセキュリティ上の考慮事項**: Fabric エンジンは接続設定者の ID で認可を行う。**Unity Catalog のガバナンスポリシーは Fabric からのアクセスには適用されない**。Unity Catalog 側で特定テーブルを公開すると、その接続にアクセスできる Fabric ユーザー全員がクエリできる。

**制限事項**: ビュー、マテリアライズドビュー、ストリーミングテーブル、行フィルター・列マスクが適用されたテーブルは非対応。**Delta 形式テーブルのみ**サポート。

**パターン 2: Databricks から OneLake にデータを書き込む**

| 認証方式 | 特徴 |
|---|---|
| Microsoft Entra ID クレデンシャルパススルー | ユーザー ID を OneLake に引き継ぐ。**Premium ワークスペース**と ADLS クレデンシャルパススルー有効化が必要。サーバーレスコンピュートは非対応 |
| サービスプリンシパル | 固定認証情報。通常のクラスターとサーバーレス両対応 |

接続パスの形式: `abfss://myWorkspace@onelake.dfs.fabric.microsoft.com/myLakehouse.lakehouse/Files/`

---

### 3-2. Power BI との統合

**3 つの統合方式**

| 方式 | 適したシナリオ |
|---|---|
| **Power BI Desktop 接続** | データアナリストが対話的にレポートを作成する場合 |
| **Databricks から Power BI サービスへ発行** | データエンジニアが管理するデータセットを Power BI ユーザーと共有する場合 |
| **Power BI タスクでの自動化** | アップストリームの変換処理完了後に自動でセマンティックモデルを更新する場合 |

**認証方式の比較**

| 認証方式 | Unity Catalog の権限適用 | 監査の粒度 | 推奨ユースケース |
|---|---|---|---|
| Microsoft Entra ID（SSO） | ユーザーごとに適用（行フィルター・列マスク有効） | ユーザーレベル | 対話的分析、ユーザー固有のガバナンスが必要な場合 |
| Import モードまたは SSO なし | 固定認証情報。全ユーザーが同じデータを参照 | 認証情報オーナーのみ | 全ユーザーに同一データを見せる場合、Power BI 内で行レベルセキュリティを実装する場合 |
| サービスプリンシパル | サービスプリンシパルの権限で適用 | サービスプリンシパルのみ | 自動化ワークフロー、スケジュールされた更新 |
| 個人アクセストークン | 単一ユーザー ID | トークンオーナーのみ | 開発・テスト（本番の自動化には非推奨） |

> **試験ポイント**: Power BI サービスへの発行には **Unity Catalog** と **Power BI Premium ライセンス**が必要。DirectQuery モードで SSO（Microsoft Entra ID）を使用すると、Unity Catalog の細かい権限制御が Power BI レポートのビューアーにも適用される。

---

### 3-3. VS Code との統合

Databricks Extension for VS Code を使うことで、ローカル開発環境からリモートの Azure Databricks クラスター上でコードを実行できる。

**主な機能**

| 機能 | 内容 |
|---|---|
| ローカル開発・リモート実行 | `.py` ファイルやノートブックをローカルで編集し、リモートクラスターで実行 |
| Databricks Connect によるデバッグ | VS Code のデバッガーでブレークポイントを設定し、リモートクラスター上のコードをステップ実行 |
| Asset Bundles によるデプロイ | Lakeflow Jobs・Lakeflow SDP・MLOps スタックを YAML 設定ファイルで定義し、環境（開発/ステージング/本番）にデプロイ |
| コード同期 | ローカルの変更をワークスペースフォルダーに自動同期（一方向） |

> **試験ポイント**: Asset Bundles はコードと設定（ジョブスケジュール、クラスターサイズ、タスク依存関係など）を Git で一元管理し、CI/CD パターンを適用できる。

## モジュール 4: Select and Configure Compute in Azure Databricks

### 学習目標
- Azure Databricks のワークロードに適したコンピュートタイプを選択できる
- ノードタイプ、オートスケール、終了設定などのパフォーマンス設定を構成できる
- Photon アクセラレーションを有効化し、適切な Databricks Runtime バージョンを選択できる
- コンピュートのアクセス権限と専用グループアクセスモードを設定できる
- 各種方法でコンピュートにライブラリをインストール・管理できる

---

### 4-1. コンピュートタイプの選択

**コンピュートタイプ比較**

| コンピュートタイプ | 推奨ユースケース | 起動時間 | 管理オーバーヘッド | コスト効率 | 主な制限 |
|---|---|---|---|---|---|
| **サーバーレス** | 対話的開発、ETL ジョブ、BI ワークロード | 2〜6 秒 | 最小（フルマネージド） | 高（ゼロへのスケールダウン、使用分のみ課金） | RDD API、R 言語、JAR ライブラリ不可 |
| **クラシック（Standard モード）** | 協調データエンジニアリング、共有分析 | 3〜7 分 | 中程度 | 中程度（マルチユーザー共有でコスト削減） | Unity Catalog によるガバナンスが必要 |
| **クラシック（Dedicated モード）** | RDD ワークロード、GPU ジョブ、R 言語、カスタムコンテナ | 3〜7 分 | 中程度 | 低（単一ユーザー/グループのみ） | 共有リソースより高コスト |
| **インスタンスプール** | 頻繁にクラシックコンピュートを起動するワークロード | 1 分未満 | 高（アイドル容量の維持が必要） | 変動（アイドルコストと使用頻度のトレードオフ） | アイドル中も VM コストが発生 |
| **SQL ウェアハウス（サーバーレス）** | SQL 分析、BI ダッシュボード、レポーティング | 2〜6 秒 | 最小（インテリジェントワークロード管理） | 高（Photon + Predictive IO、動的スケーリング） | SQL ワークロードのみ |
| **SQL ウェアハウス（Pro）** | カスタムネットワーク、フェデレーションが必要な SQL | 約 4 分 | 中程度 | 中程度 | サーバーレスより起動が遅い |
| **SQL ウェアハウス（Classic）** | 入門レベルの SQL 探索 | 約 4 分 | 中程度 | 低（基本的な Photon のみ） | パフォーマンス機能が限定的 |
| **ジョブコンピュート（サーバーレス）** | 自動化ワークフロー、本番 ETL | 2〜6 秒 | 最小（ジョブ完了後に自動終了） | 高（アイドルコストなし） | サーバーレスコンピュートと同様の制限 |

**クラシックコンピュートのクラスターモード比較**

| モード | 構成 | 適したワークロード |
|---|---|---|
| **マルチノード** | ドライバー + 複数ワーカー | 大規模データ処理、高スループットが必要な本番ワークロード |
| **シングルノード** | ドライバーのみ（ワーカーなし） | 軽量な探索、小規模データ分析、scikit-learn などの ML フレームワーク |

> **試験ポイント**: サーバーレスコンピュートは RDD API・R 言語・JAR ライブラリに**非対応**。これらが必要な場合は Dedicated アクセスモードのクラシックコンピュートを使用。

---

### 4-2. コンピュートパフォーマンスの設定

**ノードタイプの選択**

| インスタンスファミリー | 特徴 | 適したワークロード |
|---|---|---|
| **メモリ最適化（E シリーズ）** | 高いメモリ/コア比 | 大規模なジョイン、集計、インメモリ分析 |
| **コンピュート最適化（F シリーズ）** | 高い CPU パフォーマンス、低メモリ比 | 複雑な計算を伴う ETL |
| **ストレージ最適化（L シリーズ）** | 高速 NVMe ローカルストレージ | 繰り返しデータ読み取り、キャッシュを活用した分析 |
| **GPU 対応（NC/ND シリーズ）** | NVIDIA GPU 搭載 | 機械学習、ディープラーニング、画像処理（Photon は無効にする必要あり） |

**オートスケール**

最小〜最大ワーカー数を設定し、ワークロードに応じて自動調整。Databricks の最適化オートスケールは：
- スケールアップ: 2 ステップで最小から最大まで素早く拡張
- スケールダウン: アイドル状態でなくてもシャッフルファイルの状態を監視して縮小判断
- ジョブコンピュート: 40 秒ごとに評価
- 汎用コンピュート: 150 秒ごとに評価

**スポットインスタンス（Azure Spot VM）の活用**

- ワーカーノードにはスポットインスタンスを使用（最大 90% コスト削減）
- **ドライバーノードはオンデマンドインスタンス**を使用（ドライバーが失われるとクラスター全体が停止するため）
- 優先解除通知の 30 秒前から Databricks がシャッフルデータを移行するデコミッショニングを有効化するとタスク失敗を軽減

---

### 4-3. コンピュート機能の設定

**Photon アクセラレーション**

| 項目 | 内容 |
|---|---|
| 概要 | Apache Spark コンポーネントをネイティブコードで置き換えた高速クエリ実行エンジン |
| 効果的なワークロード | 大規模なジョイン、集計、テーブルスキャン、ディスクアクセスが頻繁な処理 |
| 効果が少ないワークロード | 小規模データの単純なバッチ ETL（2 秒未満で完了するような処理） |
| デフォルト | Databricks Runtime 9.1 LTS 以上で有効 |
| 制約 | **GPU 対応クラスターでは無効**にする必要あり |

**Databricks Runtime の選択**

| 用途 | 推奨 |
|---|---|
| 対話的開発（汎用コンピュート） | 最新バージョン（最新の最適化・互換性を活用） |
| 本番ジョブ（ジョブコンピュート） | **LTS（長期サポート）バージョン**（安定性重視） |
| 機械学習ワークロード | **Databricks Runtime ML**（ML ライブラリ、GPU ドライバー、CUDA フレームワークを事前インストール済み） |

## モジュール 5: Create and organize objects in Unity Catalog

### 学習目標
- Unity Catalog オブジェクトへの効果的な命名規則を適用できる
- 環境をまたいでデータアセットを整理するカタログとスキーマを作成できる
- マネージドテーブル・外部テーブル・ビュー・ボリュームを作成できる
- 関数やストアドプロシージャなどの DDL 操作を実装できる
- フォーリンカタログを使って外部データベースシステムにアクセスできる
- AI/BI Genie の指示を設定してデータの検索性を高められる

---

### 5-1. カタログの作成と整理

**カタログの役割**

カタログは Unity Catalog の 3 層名前空間のトップレベルコンテナ。データ分離の主要単位として機能し、独自のストレージロケーション・セキュリティ境界・アクセスポリシーを持てる。

**カタログ設計パターン**

| パターン | 例 | 用途 |
|---|---|---|
| **環境ベースの分離** | `dev_catalog`, `staging_catalog`, `prod_catalog` | 開発・テスト・本番の分離。本番データへの誤アクセスを防止 |
| **機密性ベースの分離** | `public_data`, `customer_pii`, `financial_data` | コンプライアンス要件・プライバシー規制への対応 |

**カタログの作成**

```sql
CREATE CATALOG IF NOT EXISTS prod_catalog
MANAGED LOCATION 'abfss://prod-data@mystorageaccount.dfs.core.windows.net/catalog-root'
COMMENT '本番カタログ（専用 ADLS Gen2 ストレージ）';
```

マネージドストレージを指定しない場合、マネージドテーブルとボリュームはメタストアのデフォルトストレージに配置される（組織ポリシーに合わない場合があるため、**カタログレベルのストレージ指定を推奨**）。

**ワークスペースバインディング**: デフォルトでは新規カタログはメタストアにアタッチされた全ワークスペースからアクセス可能。本番カタログは本番ワークスペースのみに制限することを推奨。

---

### 5-2. テーブルとビューの作成

**テーブルの種類比較（試験の重要ポイント）**

| 項目 | マネージドテーブル | 外部テーブル |
|---|---|---|
| ストレージ管理 | Unity Catalog が管理 | 外部ストレージで管理 |
| データ形式 | Delta Lake（デフォルト） | Delta, CSV, JSON, Parquet 等の複数形式 |
| テーブル削除時 | 8 日間の保持期間後にデータファイルも削除 | メタデータのみ削除、データは外部ストレージに残存 |
| 推奨ケース | 新規 ETL パイプライン、Azure Databricks 内でデータを完結させる場合 | 他システムとデータを共有する場合、既存ストレージのデータをガバナンス対象にする場合 |

**ビューの種類比較**

| 種類 | データ保持 | 更新タイミング | 適したユースケース |
|---|---|---|---|
| **標準ビュー** | なし（クエリのたびに計算） | 自動（常に最新データ） | 複雑なジョインのカプセル化、頻繁に変わるデータの参照 |
| **動的ビュー** | なし（クエリのたびに計算） | 自動 | 行レベルフィルター・列レベルマスクによる細かいアクセス制御 |
| **マテリアライズドビュー** | あり（Delta テーブルとしてキャッシュ） | 手動またはスケジュール | 重い集計処理、繰り返しアクセスされる分析クエリ |

**動的ビューによる列マスクの例**

```sql
CREATE VIEW sales_redacted AS
SELECT
  user_id,
  CASE WHEN
    is_account_group_member('auditors') THEN email
    ELSE 'REDACTED'
  END AS email,
  country_region, product, total
FROM sales_raw;
```

**マテリアライズドビューの増分リフレッシュ**（増分リフレッシュには行トラッキングの有効化が必要）

```sql
ALTER TABLE sales_transactions
SET TBLPROPERTIES (delta.enableRowTracking = true);  -- Runtime 14.1 以上が必要
```

> **試験ポイント**: 動的ビューは SQL ウェアハウス、Standard アクセスモード、または Dedicated アクセスモード（Runtime 15.4 LTS 以上）でのみ使用可能。

**液体クラスタリング（Liquid Clustering）**

従来のパーティショニングに代わる推奨手法。`CLUSTER BY` 句でクエリパターンに基づいてデータレイアウトを自動最適化。マニュアルでのパーティション設定が不要で、スキューが発生しにくい。

---

### 5-3. ボリュームの作成

ボリュームはテーブル化されていないファイル（CSV、JSON、画像、ML アーティファクトなど）をクラウドオブジェクトストレージで管理するための Unity Catalog オブジェクト。

**マネージドボリューム vs 外部ボリューム**

| 項目 | マネージドボリューム | 外部ボリューム |
|---|---|---|
| ストレージ場所 | Unity Catalog が自動管理 | 作成時に指定（既存のクラウドストレージ） |
| 削除時の挙動 | 7 日間の保持期間後にファイルを削除 | Unity Catalog の登録のみ削除、ファイルは残存 |
| 推奨ケース | Azure Databricks 内でファイルを完結させる場合 | 他システムとファイルを共有する場合、既存ストレージへのガバナンス追加 |

**アクセスパス**: `/Volumes/<catalog>/<schema>/<volume>/<path>/<filename>`

このパスは Spark、pandas、SQL、dbutils.fs など全ツールで共通に使用可能。

**ボリューム権限**

| 権限 | 内容 |
|---|---|
| `READ VOLUME` | ファイルの一覧取得・読み取り |
| `WRITE VOLUME` | ファイルの作成・更新・削除 |
| `CREATE VOLUME` | スキーマ内での新規ボリューム作成 |
| `CREATE EXTERNAL VOLUME` | 外部ロケーションを使った外部ボリューム作成 |

> **注意**: ボリュームはフォルダーレベルや サブディレクトリレベルの ACL をサポートしない。権限はボリューム全体に適用される。異なるアクセス制御が必要な場合は、別々のボリュームを作成すること。

---

### 5-4. フォーリンカタログの実装

フォーリンカタログは、PostgreSQL・MySQL・Snowflake・BigQuery などの外部データベースシステムのデータを **ETL なしで**直接クエリしながら、Unity Catalog のガバナンスを適用する仕組み。

**実装手順**

1. **接続（Connection）の作成**: 外部データベースへの接続情報（ホスト、ポート、認証情報）を Unity Catalog に登録
2. **フォーリンカタログの作成**: 接続と外部データベースを指定してカタログを作成
3. **権限の付与**: 標準の Unity Catalog 権限モデルでアクセス制御
4. **クエリの実行**: 通常の `catalog.schema.table` 形式でクエリ

```sql
-- ステップ1: 接続の作成
CREATE CONNECTION postgresql_prod TYPE postgresql
OPTIONS (
  host 'prod-db.example.com',
  port '5432',
  user secret ('prod-secrets','postgres-user'),
  password secret ('prod-secrets','postgres-password')
);

-- ステップ2: フォーリンカタログの作成
CREATE FOREIGN CATALOG IF NOT EXISTS prod_customer_data
USING CONNECTION postgresql_prod
OPTIONS (database 'customers');

-- メタデータの手動更新（高頻度アクセス時に有効）
REFRESH FOREIGN CATALOG prod_customer_data;
```

**フォーリンカタログの利点**

- **データ移行なし**: データをその場でクエリ（POC 分析や移行前の検証に最適）
- **一元ガバナンス**: 外部データへのアクセスも Unity Catalog の監査ログに記録
- **段階的移行のサポート**: レガシーシステムのデータを維持しつつ、段階的に Unity Catalog へ移行可能

## まとめ: 重要ポイント

### コンポーネントと役割の対応

| コンポーネント | 役割 | 管理者 |
|---|---|---|
| アカウント | 組織全体の Azure Databricks 管理 | 管理者 |
| メタストア | Unity Catalog のトップレベルメタデータコンテナ（リージョン単位） | 管理者 |
| ワークスペース | 協調作業環境、コンピュートワークロードの実行 | 管理者/ユーザー |
| コントロールプレーン | Web アプリ、ジョブスケジューリング、クラスター管理 | Databricks（Microsoft）|
| クラシックコンピュートプレーン | 実データ処理（ユーザーの Azure サブスクリプション内） | ユーザー |
| サーバーレスコンピュートプレーン | 実データ処理（Databricks の Azure アカウント内） | Databricks |

### 試験で問われる主な比較ポイント

| 比較テーマ | 選択基準 |
|---|---|
| **サーバーレス vs クラシックコンピュート** | RDD/R/JAR が必要 → クラシック。それ以外 → サーバーレスが優先 |
| **Standard vs Dedicated アクセスモード** | マルチユーザー共有 → Standard。RDD/GPU/R/カスタムコンテナ → Dedicated |
| **マネージドテーブル vs 外部テーブル** | Databricks 完結 → マネージド。他システムとデータ共有/既存ストレージ → 外部 |
| **マネージドボリューム vs 外部ボリューム** | Databricks 完結 → マネージド。他システムとファイル共有 → 外部 |
| **標準ビュー vs マテリアライズドビュー** | リアルタイムデータ必要/軽量クエリ → 標準。重い集計/繰り返しアクセス → マテリアライズド |
| **Photon 有効 vs 無効** | 大規模ジョイン・集計 → 有効。GPU クラスター・小規模処理 → 不要または無効 |
| **SQL ウェアハウス サーバーレス vs Pro vs Classic** | 一般的な SQL/BI → サーバーレス。カスタムネットワーク → Pro。基本的な探索 → Classic |
| **Fabric 統合: Unity Catalog 読み取り vs OneLake 書き込み** | Unity Catalog 既存データを Fabric に公開（セキュリティ要件確認必須）→ 読み取り。Databricks で加工済みデータを Fabric に送る → 書き込み |

### スコープ比較

| 項目 | Hive メタストア | Unity Catalog |
|---|---|---|
| スコープ | ワークスペース単位 | アカウントレベル（複数ワークスペースで共有） |
| リネージ | 限定的 | 列レベルまで追跡 |
| 推奨 | レガシー | 現在の標準（移行を推奨） |
