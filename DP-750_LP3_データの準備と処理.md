## モジュール 1: Design and implement data modeling with Azure Databricks

### 学習目標
- データ取り込みロジックを設計し、データソース接続を構成する
- シナリオに応じた適切なデータ取り込みツールを選択する
- Delta Lake、Apache Iceberg などのテーブルフォーマットを比較・選択する
- 効果的なデータパーティショニングスキームを設計・実装する
- SCD（緩やかに変化するディメンション）タイプを選択・実装する
- 変更追跡と監査のためのテンポラルテーブルを設計・実装する
- ファクトテーブルとディメンションテーブルに適切なデータ粒度を選択する
- クエリ最適化のためのクラスタリング戦略を設計・実装する
- マネージドテーブルとアンマネージドテーブルの使い分けを判断する

---

### 1-1. テーブルフォーマットの選択

Azure Databricks では複数のオープンテーブルフォーマットがサポートされている。デフォルトは Delta Lake。

| フォーマット | 特徴 | 推奨シナリオ |
|---|---|---|
| **Delta Lake** | Azure Databricks のデフォルト。ACID トランザクション、タイムトラベル、スキーマ強制、CDC をネイティブサポート | Databricks 中心のワークフロー全般 |
| **Apache Iceberg** | オープン標準の分析テーブルフォーマット。スキーマ進化、タイムトラベル、隠れパーティショニングをサポート | マルチエンジン環境（Snowflake、Spark、Trino との連携） |
| **Parquet（外部）** | 生の Parquet ファイル。トランザクション管理なし | 読み取り専用の外部データアクセス |

> **試験ポイント**: Unity Catalog 管理の Delta Lake テーブルと Iceberg テーブルは、パフォーマンスと運用上の優位性から**デフォルトかつ推奨されるテーブルタイプ**。Delta Uniform を使うと、Delta テーブルのまま Iceberg クライアントから読み取れる。

---

### 1-2. マネージドテーブル vs アンマネージドテーブル（外部テーブル）

| 項目 | マネージドテーブル | アンマネージドテーブル（外部テーブル） |
|---|---|---|
| ストレージ管理 | Unity Catalog が管理 | ユーザーが管理（ADLS など） |
| テーブル削除時 | データも削除される | メタデータのみ削除（データは残る） |
| 最適化 | 自動最適化・コスト削減 | Unity Catalog はガバナンスのみ担当 |
| 推奨度 | **推奨**（デフォルト） | 既存データの登録、他サービスとの共有時 |

---

### 1-3. データパーティショニング戦略

| 戦略 | 概要 | 推奨シナリオ |
|---|---|---|
| **パーティション列** | 特定の列（日付など）でデータを物理分割 | 1 TB 超の大規模テーブル、フィルタ条件が固定の場合 |
| **液体クラスタリング（Liquid Clustering）** | Databricks が動的にクラスタリングキーを管理。増分処理のみ書き換え | **全ての新規 Delta テーブルに推奨**（1 TB 未満は特にこちら） |
| **Z-order** | 複数列の多次元クラスタリング。手動実行が必要 | 液体クラスタリング移行前のレガシー最適化 |

> **試験ポイント**: Databricks は全ての新規テーブルに**液体クラスタリングを推奨**。`CLUSTER BY AUTO` を使うとクエリワークロードに基づいてクラスタリングキーを自動選択する。従来のパーティション + Z-order の組み合わせより優れている。

---

### 1-4. SCD（緩やかに変化するディメンション）タイプ

| タイプ | 動作 | データ保持 | 実装方法 |
|---|---|---|---|
| **SCD Type 1** | 既存レコードを上書き | 現在値のみ | `AUTO CDC INTO ... STORED AS SCD TYPE 1` |
| **SCD Type 2** | 新レコードを追加して履歴を保持 | 全履歴（`__START_AT`/`__END_AT` 列） | `AUTO CDC INTO ... STORED AS SCD TYPE 2` |

AUTO CDC API（旧 APPLY CHANGES API）は SCD Type 1/2 の複雑な計算を自動化する。Databricks は最新の **AUTO CDC API** の使用を推奨している。

SCD Type 2 のテーブルスキーマには、シーケンス列と同じデータ型の `__START_AT` と `__END_AT` 列が必要。

---

### 1-5. テンポラルテーブルと変更追跡

テンポラルテーブルは変更の完全な履歴を記録し、監査やポイントインタイムクエリを可能にする。Delta Lake のタイムトラベル機能（`VERSION AS OF`、`TIMESTAMP AS OF`）を活用することで、任意の時点のデータを参照できる。

---

### 1-6. データ粒度の選択

| 粒度 | 内容 |
|---|---|
| **ファクトテーブル** | 最も細かい粒度（トランザクションレベル）が基本。集計後は情報が失われる |
| **ディメンションテーブル** | エンティティごとに 1 行（SCD で履歴管理） |
| **集計テーブル** | 集計済みのサマリーデータ。クエリ高速化目的 |

粒度の選択は、ダウンストリームのユースケースとクエリパターンに基づいて決定する。

## モジュール 2: Ingest data into Unity Catalog

### 学習目標
- Lakeflow Connect を使ってマネージドコネクタで外部ソースからデータを取り込む
- ノートブックで DataFrame と Structured Streaming を使ってバッチ・ストリーミングデータを取り込む
- `COPY INTO` と `CREATE TABLE AS SELECT` でファイルベースのバッチ取り込みを行う
- AUTO CDC API で変更データキャプチャフィードを処理する
- Kafka / Event Hubs からリアルタイムデータを Spark Structured Streaming で取り込む
- Auto Loader で新規ファイルを自動検出し、スキーマ進化に対応する
- Lakeflow Spark Declarative Pipelines でデータ取り込みワークフローを統合管理する

---

### 2-1. データ取り込み手法の比較

| 手法 | 種類 | 主なユースケース |
|---|---|---|
| **Lakeflow Connect** | バッチ / 増分 | SaaS / DB ソースへのマネージドコネクタ（設定ベース、コード不要） |
| **ノートブック（DataFrame）** | バッチ / ストリーミング | カスタムロジックが必要な汎用取り込み |
| **COPY INTO** | バッチ | クラウドストレージからの冪等なファイル一括ロード |
| **Auto Loader** | ストリーミング / 増分 | クラウドストレージへの継続的なファイル到着を自動処理 |
| **Spark Structured Streaming** | ストリーミング | Kafka / Event Hubs などのメッセージバスからのリアルタイム取り込み |
| **AUTO CDC API** | 増分 | CDC フィードの処理（SCD Type 1/2 を自動化） |
| **Lakeflow Spark Declarative Pipelines** | バッチ / ストリーミング | 複数の取り込みパターンをパイプラインとして宣言的に管理 |

---

### 2-2. Lakeflow Connect

マネージドコネクタを使って外部データソースへの接続を管理する。コードを書かずに UI や API で設定でき、増分取り込みに対応している。

**動作の特徴:**
- 初回実行: 全データを取り込みながら変更を並行して追跡
- 2回目以降: 変更分のみを取り込む（増分処理）
- SQL Server との連携では CDC（変更データキャプチャ）と変更追跡の両方をサポート

---

### 2-3. COPY INTO vs Auto Loader

| 比較項目 | COPY INTO | Auto Loader |
|---|---|---|
| 処理方式 | バッチ（冪等、同一ファイルを再処理しない） | ストリーミング / 増分 |
| スキーマ推論 | 手動または明示的指定 | 自動（`cloudFiles.schemaEvolutionMode` で設定） |
| 推奨用途 | 定期的なバッチロード | 継続的に到着するファイルの自動処理 |
| Unity Catalog | 外部ロケーションが必要 | Unity Catalog 対応パイプライン内で外部ロケーションが必要 |

> **試験ポイント**: Databricks は**クラウドオブジェクトストレージからのほとんどの取り込みに Auto Loader をパイプライン内で使用することを推奨**している。Auto Loader はファイルの増分・冪等ロードに最適化されている。

---

### 2-4. Spark Structured Streaming

Kafka / Event Hubs などのメッセージバスからリアルタイムにデータを取り込む。Structured Streaming は小さなバッチを連続的に処理し、ストリーミングテーブルへ書き込む。

**ストリーミングテーブルの特徴:**
- Auto Loader またはメッセージバスからのデータ取り込みに適している
- Unity Catalog で管理・ガバナンスが可能
- Databricks SQL からもクエリ可能

---

### 2-5. AUTO CDC API による CDC 処理

変更データキャプチャ（CDC: Change Data Capture）フィードを処理し、SCD Type 1/2 のロジックを自動化する。

```sql
-- SCD Type 2 の例
CREATE STREAMING TABLE target_table
FLOW AUTO CDC
INTO target_table
USING source_stream
KEYS (id)
SEQUENCE BY updated_at
STORED AS SCD TYPE 2;
```

- **順序外レコードの自動処理**: シーケンス列で順序を保証
- **SCD Type 1**: レコードを上書き（現在値のみ保持）
- **SCD Type 2**: 履歴を保持（`__START_AT` / `__END_AT` 列で期間管理）

---

### 2-6. Lakeflow Spark Declarative Pipelines

複数の取り込みパターン（Auto Loader、Structured Streaming、CDC など）をパイプラインとして宣言的に定義し、スケジュール実行や依存関係を管理する。Delta Live Tables の後継として Unity Catalog との統合が強化されている。

## モジュール 3: Cleanse, transform, and load data into Unity Catalog

### 学習目標
- SQL コマンドとデータプロファイリング機能を使ってデータ品質を評価する
- ストレージ最適化とデータ整合性のために適切な列データ型を選択する
- データセット内の重複値・欠損値・null 値を特定して解決する
- フィルタリング・グルーピング・集計操作でデータを変換する
- JOIN と集合演算子（UNION、INTERSECT、EXCEPT）でデータセットを結合する
- 非正規化、ピボット、アンピボットでデータを再構造化する
- INSERT、MERGE、上書き操作で変換済みデータを Unity Catalog テーブルへロードする

---

### 3-1. データプロファイリング

データ変換の前に品質を評価するための手法。

| 手法 | 内容 |
|---|---|
| `DESCRIBE EXTENDED` | テーブルのスキーマ、統計情報を確認 |
| `SELECT COUNT(*)`, `COUNT(DISTINCT col)` | レコード数・ユニーク値の把握 |
| `SUMMARIZE` / データプロファイリング UI | 列ごとの統計量（平均、最大、最小、null 率）を可視化 |
| ヒストグラム・分布確認 | 値の分布やスキューを把握 |

Unity Catalog のデータプロファイリング機能は、テーブルのプロファイリングメトリクスを時系列で記録し、履歴トレンドの確認を可能にする。

---

### 3-2. 列データ型の選択

| 型 | 推奨用途 |
|---|---|
| `INT` / `BIGINT` | 整数値（ID、カウント） |
| `DECIMAL(p,s)` | 金額・精度が重要な数値 |
| `TIMESTAMP` / `DATE` | 日時データ（タイムゾーン考慮は TIMESTAMP） |
| `STRING` | 可変長テキスト |
| `BOOLEAN` | フラグ値 |
| `STRUCT` / `ARRAY` / `MAP` | 半構造化データ（ネスト構造） |

> **試験ポイント**: 不適切なデータ型（例: 金額を FLOAT で格納）は精度問題を招く。`DECIMAL` を使うこと。

---

### 3-3. 重複・null 値の処理

| 問題 | 対処方法 |
|---|---|
| **完全重複** | `DISTINCT` でフィルタリング |
| **部分重複（キー重複）** | `ROW_NUMBER() OVER(PARTITION BY key ORDER BY ...)` で最新レコードを選択 |
| **null 値** | `COALESCE(col, default_value)` で既定値に置換、または `WHERE col IS NOT NULL` でフィルタリング |
| **欠損値** | ドメイン知識に基づく補完（平均値、中央値、前後の値） |

---

### 3-4. フィルタリング・集計変換

```sql
-- フィルタリング + グルーピング + 集計
SELECT category, region, SUM(sales) AS total_sales, COUNT(*) AS record_count
FROM raw_sales
WHERE date >= '2024-01-01'
GROUP BY category, region
HAVING total_sales > 10000
ORDER BY total_sales DESC;
```

---

### 3-5. JOIN と集合演算子

| 演算子 | 目的 |
|---|---|
| `INNER JOIN` | 両テーブルに存在するレコードのみ結合 |
| `LEFT JOIN` | 左テーブルの全レコードを保持（右は null 許容） |
| `FULL OUTER JOIN` | 両テーブルの全レコードを保持 |
| `UNION` / `UNION ALL` | 複数クエリの結果を縦に結合（UNION は重複除去） |
| `INTERSECT` | 両クエリに共通するレコード |
| `EXCEPT` | 左クエリにあって右クエリにないレコード |

---

### 3-6. 非正規化・ピボット・アンピボット

| 手法 | 目的 |
|---|---|
| **非正規化** | JOIN を減らしてクエリパフォーマンスを向上（分析用テーブルに適用） |
| **ピボット（PIVOT）** | 行を列に変換（クロス集計表の生成） |
| **アンピボット（UNPIVOT）** | 列を行に変換（縦持ちデータへの変換） |
| **EXPLODE** | 配列・マップを複数行に展開（半構造化データの正規化） |
| **FLATTEN** | ネスト構造を平坦化 |

---

### 3-7. データロード戦略

| 戦略 | SQL | 特徴 | 推奨シナリオ |
|---|---|---|---|
| **追記（Append）** | `INSERT INTO` | 既存データを変更せず末尾に追加 | イベントログ、不変データ |
| **上書き（Overwrite）** | `INSERT OVERWRITE` / `df.write.mode("overwrite")` | テーブル全体を置き換え | スナップショット更新、小〜中規模テーブル |
| **マージ（Upsert）** | `MERGE INTO` | 条件に応じてINSERT/UPDATE/DELETEを選択 | SCD Type 1、増分更新 |

```sql
-- MERGE INTO の例
MERGE INTO target AS t
USING source AS s ON t.id = s.id
WHEN MATCHED THEN UPDATE SET t.value = s.value, t.updated_at = s.updated_at
WHEN NOT MATCHED THEN INSERT (id, value, updated_at) VALUES (s.id, s.value, s.updated_at)
WHEN NOT MATCHED BY SOURCE THEN DELETE;
```

> **試験ポイント**: `MERGE INTO` は SCD Type 1 の実装に使用するが、Lakeflow Spark Declarative Pipelines では **AUTO CDC API** の使用が推奨される。

## モジュール 4: Implement and manage data quality constraints with Azure Databricks

### 学習目標
- null 許容性・カーディナリティ・範囲制約のバリデーションチェックを実装する
- スキーマ強制と明示的なキャストによってデータ型チェックを実装する
- Auto Loader と Delta Lake を使ってスキーマ強制とスキーマドリフトを管理する
- Lakeflow Spark Declarative Pipelines のパイプラインエクスペクテーションでデータ品質を管理する

---

### 4-1. バリデーションチェックの実装

| 制約タイプ | 実装方法 | 例 |
|---|---|---|
| **null 非許容** | `NOT NULL` 制約、または `WHERE col IS NOT NULL` | 必須フィールドの検証 |
| **カーディナリティ** | `COUNT(DISTINCT col) / COUNT(*)` で一意性を確認 | 主キーの重複チェック |
| **範囲制約** | `WHERE col BETWEEN min AND max` | 年齢が 0〜150 の範囲内かチェック |
| **参照整合性** | ファクトテーブルのキーがディメンションテーブルに存在するか確認 | 孤立レコードの検出 |

Delta テーブルのテーブル制約（`CHECK` 制約）を使って、書き込み時にデータ品質を強制することもできる。

---

### 4-2. データ型チェックとスキーマ強制

| 手法 | 説明 |
|---|---|
| **スキーマ強制（Schema Enforcement）** | Delta Lake は書き込み時にテーブルスキーマと一致しないデータを自動的に拒否する |
| **明示的キャスト** | `CAST(col AS INT)` で型変換を強制。変換失敗時は null を返す |
| **TRY_CAST** | 変換失敗時に例外を発生させず null を返す安全なキャスト |

---

### 4-3. スキーマドリフトの検出と管理

スキーマドリフトとは、ソースデータのスキーマが予告なく変化する現象。

| 手法 | 説明 |
|---|---|
| **Auto Loader のスキーマ進化** | `cloudFiles.schemaEvolutionMode` で自動対応を設定 |
| **rescuedDataColumn** | スキーマに合致しない列を JSON 形式で別列に保存（データを失わない） |
| **Delta Lake のスキーマ進化** | `mergeSchema` オプションで列の追加を許可 |

**Auto Loader の `schemaEvolutionMode` 設定:**

| モード | 動作 |
|---|---|
| `addNewColumns`（デフォルト） | 新しい列を自動追加 |
| `rescue` | 新しい列を `_rescued_data` に保存 |
| `failOnNewColumns` | 新しい列が検出されたらパイプラインを失敗させる |
| `none` | スキーマの変化を無視 |

> **試験ポイント**: Auto Loader は**列の追加には対応するが、型変更には対応しない**。型変更が検出された場合はパイプラインが失敗する。

---

### 4-4. パイプラインエクスペクテーション

Lakeflow Spark Declarative Pipelines のエクスペクテーションは、データ品質制約を宣言的に定義してパイプライン内で適用する機能。

**エクスペクテーションの構成要素:**
1. **説明**: 制約の名前
2. **不変条件（invariant）**: データが満たすべき条件式
3. **違反時のアクション（違反ポリシー）**: 制約違反レコードの扱い

**違反ポリシーの比較:**

| ポリシー | 動作 | 違反レコードの扱い | 推奨シナリオ |
|---|---|---|---|
| `WARN`（デフォルト） | パイプラインは継続し、メトリクスに記録 | ターゲットテーブルに書き込まれる | 全データを保持しつつ品質を可視化したい場合 |
| `DROP` | パイプラインは継続し、違反レコードを破棄 | 書き込まれない | 不正レコードが下流に伝播してはいけない場合 |
| `FAIL` | パイプラインが即座に失敗 | 書き込まれない（メトリクスなし） | 不正データが重大な問題を示す場合（厳格な品質保証） |

```python
# Python API でのエクスペクテーション例

@dlt.expect_or_drop("non_null_id", "id IS NOT NULL")
```

> **試験ポイント**: `WARN` ではパイプライン UI でメトリクスを確認できるが、`FAIL` では失敗時にメトリクスが記録されない。`DROP` と `WARN` のメトリクスはパイプライン UI で追跡可能。

## まとめ: 重要ポイント

### 試験で頻出の比較ポイント

| テーマ | 選択肢A | 選択肢B | 判断基準 |
|---|---|---|---|
| テーブルフォーマット | Delta Lake | Apache Iceberg | Databricks 中心 → Delta Lake。マルチエンジン → Iceberg（Delta Uniform 経由も可） |
| テーブル管理 | マネージドテーブル | 外部テーブル（アンマネージド） | 基本はマネージドを推奨。既存データや外部共有が必要な場合は外部テーブル |
| クラスタリング | 液体クラスタリング | Z-order + パーティション | 全ての新規テーブルは液体クラスタリングを推奨 |
| SCD | Type 1（上書き） | Type 2（履歴保持） | 現在値のみ必要 → Type 1。変更履歴が必要 → Type 2 |
| バッチ取り込み | COPY INTO | Auto Loader | 単発バッチ → COPY INTO。継続的なファイル到着 → Auto Loader |
| データロード | INSERT INTO | MERGE INTO | 追記のみ → INSERT INTO。更新・削除を含む → MERGE INTO |
| 品質制約違反 | WARN | DROP | FAIL | 全データ保持+可視化 → WARN。下流保護 → DROP。厳格保証 → FAIL |
| スキーマドリフト | addNewColumns | failOnNewColumns | 新列を許容 → addNewColumns。厳格管理 → failOnNewColumns |

### モジュール間の全体フロー

```
データモデリング設計          データ取り込み               変換・クレンジング          品質制約
(モジュール1)                (モジュール2)                (モジュール3)              (モジュール4)
─────────────────────────────────────────────────────────────────────────────────────────────
テーブルフォーマット選択  →  Lakeflow Connect             プロファイリング      →   バリデーションチェック
パーティション/クラスタ  →  Auto Loader / COPY INTO  →   重複/null 処理        →   スキーマ強制
SCD タイプ決定          →  CDC / Structured Streaming →  JOIN / 集計変換       →   パイプラインエクスペクテーション
マネージド/外部テーブル  →  Lakeflow Declarative Pipelines → MERGE / INSERT     →   スキーマドリフト管理
```
