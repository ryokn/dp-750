## モジュール 1: データパイプラインの設計と実装

### 学習目標

- インジェストからサービングまでのパイプライン処理順序を設計できる
- ユースケースに応じてノートブックと Lakeflow Spark Declarative Pipelines を使い分けられる
- Lakeflow Jobs のタスクロジック・依存関係・実行パターンを設計できる
- リトライポリシーと Expectations を含むエラーハンドリング戦略を実装できる
- ノートブックと Lakeflow Spark Declarative Pipelines の両方でパイプラインを作成できる

---

### 1-1. パイプラインの処理順序設計

データパイプラインの設計では、データの流れをレイヤーで整理する **Medallion アーキテクチャ** が標準的なパターンとして採用される。

```
[データソース] → ブロンズ（生データ） → シルバー（検証・クリーニング済み） → ゴールド（集計・分析用）
```

| レイヤー | 役割 | データの状態 |
|---------|------|------------|
| **ブロンズ** | 生データの取り込み（ランディングゾーン） | 未加工・スキーマなし |
| **シルバー** | データのクリーニング・検証・結合 | 検証済み・正規化済み |
| **ゴールド** | ビジネスロジックの集計・分析ビュー | 集計済み・レポート向け |

**設計の重要ポイント**: 各レイヤーの責務を明確に分離することで、パイプラインの保守性と再利用性が向上する。ゴールド層のテーブルは、下流の BI ツールや ML モデルへのサービングを想定して設計する。

---

### 1-2. ノートブック vs Lakeflow Spark Declarative Pipelines の選択

どちらを選ぶかは、要件の複雑さと開発の生産性のバランスによって決まる。

| 観点 | ノートブック | Lakeflow Spark Declarative Pipelines |
|------|------------|--------------------------------------|
| **コードスタイル** | 手続き型（PySpark / SQL） | 宣言型（Python / SQL） |
| **オーケストレーション** | Lakeflow Jobs で手動管理 | 依存関係を自動解析・自動実行 |
| **エラーハンドリング** | 手動でリトライ実装が必要 | タスク→フロー→パイプラインの段階的リトライを自動処理 |
| **データ品質** | 手動でバリデーション実装 | Expectations による宣言的なデータ品質チェック |
| **CDC 対応** | 手動実装が複雑 | AUTO CDC API で SCD Type 1/2 を宣言的に処理 |
| **適したシナリオ** | 複雑なカスタムロジック、サポート外のデータソース | インクリメンタル取り込み、ストリーミング変換、CDC |

**試験のポイント**: Lakeflow Spark Declarative Pipelines（旧 Delta Live Tables）は、手続き型の Spark コードを大幅に削減し、自動オーケストレーションと組み込みデータ品質機能を提供する。複数環境で一貫した再現性が求められる場合や、複雑な依存関係を持つパイプラインには Lakeflow SDP が優位。

---

### 1-3. Lakeflow Spark Declarative Pipelines の主要概念

#### データセット型

| データセット型 | 処理方式 | 主な用途 |
|--------------|---------|---------|
| **ストリーミングテーブル** | ストリーミング（各レコードを一度だけ処理） | データ取り込み、低レイテンシー変換、追記専用ワークロード |
| **マテリアライズドビュー** | バッチ（インクリメンタルリフレッシュ） | 集計・変換、変更があったデータのみ再処理 |
| **一時ビュー** | バッチ（中間処理用） | パイプライン内の中間テーブル（外部から参照不可） |
| **シンク（Sink）** | ストリーミング | Delta 以外の出力先（Kafka、EventHubs など） |

#### フロー型

- **Append フロー**: 新規レコードを追記するストリーミングフロー
- **AUTO CDC フロー**: CDC イベントの順序外処理・重複排除を自動処理。SCD Type 1（上書き）と SCD Type 2（履歴保持）の両方をサポート
- **マテリアライズドビューフロー**: バッチフロー。変更差分のみ処理するインクリメンタルエンジンを使用

---

### 1-4. エラーハンドリング戦略

#### Lakeflow SDP の Expectations（データ品質ルール）

Expectations は、パイプラインのデータ品質を宣言的に定義する仕組み。

| 違反時の動作 | 説明 | ユースケース |
|------------|------|------------|
| `WARN`（警告） | 違反レコードを含めて処理を継続し、メトリクスを記録 | 品質監視・レポート目的 |
| `DROP`（削除） | 違反レコードをドロップして処理を継続 | 品質の低いデータを除外したい場合 |
| `FAIL`（失敗） | パイプライン全体を停止 | データ品質が業務クリティカルな場合 |

#### リトライポリシー（Lakeflow Jobs）
ノートブックベースのパイプラインでは、Lakeflow Jobs のリトライポリシーでエラーに対処する。

- **タスクレベルのリトライ**: 特定タスクの失敗時に再試行
- **最大リトライ回数**: タスクごとに設定可能
- **リトライ間隔**: 指数バックオフまたは固定間隔で設定

## モジュール 2: Lakeflow Jobs の実装

### 学習目標

- タスクとコンピュートリソースを含む Lakeflow Jobs を作成・設定できる
- テーブル更新やファイル到着を含むジョブトリガーを設定できる
- インターバルと cron 式を使用してジョブをスケジューリングできる
- 監視のためのジョブアラートと通知を設定できる
- 信頼性確保のための自動再起動とリトライポリシーを設定できる

---

### 2-1. Lakeflow Jobs の基本構造

Lakeflow Jobs は、Azure Databricks における主要なオーケストレーション機能。複数のタスクを定義し、それらの依存関係・実行順序・コンピュートリソースを一元管理する。

**ジョブの構成要素:**

| 要素 | 説明 |
|------|------|
| **タスク** | 実行する処理の単位（ノートブック、Python スクリプト、SQL、Lakeflow Pipeline など） |
| **依存関係** | タスク間の実行順序（上流タスクの成否に基づく制御フロー） |
| **クラスター設定** | 各タスクで使用するコンピュートリソース（ジョブクラスター or 既存クラスター） |
| **パラメーター** | タスクへの入力値（ウィジェット経由またはタスクパラメーター） |

---

### 2-2. トリガーの種類

| トリガー種別 | 説明 | ユースケース |
|------------|------|------------|
| **スケジュール（Cron）** | 時刻ベースで定期実行 | 定期バッチ処理 |
| **ファイル到着トリガー** | クラウドストレージへのファイル到着を検知して実行 | イベント駆動の取り込みパイプライン |
| **テーブル更新トリガー** | Unity Catalog のテーブルが更新された際に実行 | データ連鎖パイプライン |
| **手動実行** | UI または API から即時実行 | テスト・アドホック実行 |
| **継続実行（Continuous）** | 直前の実行完了後に即座に再開 | 擬似リアルタイム処理 |

**試験のポイント**: ファイル到着トリガーとテーブル更新トリガーはイベント駆動アーキテクチャに対応するための重要な機能。スケジュール実行との違い（時刻ベース vs イベントベース）を押さえておくこと。

---

### 2-3. スケジューリング

スケジュールは **Quartz cron 式**または**インターバル指定**で設定する。

```
# Quartz cron 式の例（毎日午前 2 時に実行）
0 0 2 * * ?

# フィールドの順序: 秒 分 時 日 月 曜日 年（オプション）
```

**インターバル指定**: 分・時間・日単位での繰り返し実行を簡単に設定できる（UI から選択）。

---

### 2-4. アラートと通知

ジョブの状態変化（成功・失敗・タイムアウトなど）に応じて通知を送信できる。

| 通知先 | 設定方法 |
|--------|---------|
| **メール** | ジョブ設定でメールアドレスを直接指定 |
| **Slack / Teams** | Webhook URL を通知先に設定 |
| **通知先プリセット** | ワークスペース管理者が設定した通知先を再利用 |

通知をトリガーするイベント:
- ジョブ開始・成功・失敗
- タスク失敗・ストリーミング開始/停止
- 実行時間超過（SLA 違反）

---

### 2-5. 自動再起動とリトライポリシー

| 設定項目 | 説明 |
|---------|------|
| **最大リトライ回数** | タスク失敗時に再試行する最大回数（0 = リトライなし） |
| **リトライ間隔** | 再試行までの待機時間（分単位） |
| **タイムアウト** | タスクがこの時間を超えた場合に強制失敗 |
| **実行の修復（Repair Run）** | 失敗したジョブ実行を、成功済みタスクをスキップして再実行 |

**Repair Run の重要性**: 長時間のジョブで途中のタスクが失敗した場合、成功済みタスクを再実行せずに失敗タスクから再開できるため、コストと時間を大幅に削減できる。これは試験でも頻出のトピック。

## モジュール 3: 開発ライフサイクルプロセスの実装

### 学習目標

- Git フォルダーを使用した Git バージョン管理のベストプラクティスを適用できる
- ブランチ管理、プルリクエスト、競合解決を行える
- ユニット・統合・E2E・UAT を含むテスト戦略を実装できる
- デプロイ自動化のために Databricks Asset Bundles を設定・カスタマイズできる
- Databricks CLI を使用して開発・本番環境へバンドルをデプロイできる

---

### 3-1. Git フォルダーによるバージョン管理

Azure Databricks の **Git フォルダー**は、Azure DevOps・GitHub・GitLab などのリモートリポジトリとワークスペースを同期させる機能。

**Git フォルダーの特徴:**
- ノートブックや Python ファイルをソース管理下に置ける
- ブランチの切り替え・プル・コミットを UI から操作可能
- 複数人での並行開発に対応

**制限事項（重要）**: Git フォルダーはコードファイル（ノートブック、ダッシュボードドラフト）のみをソース管理下に置く。ジョブやパイプラインの**設定情報はソース管理対象外**。設定も含めた完全な管理には Databricks Asset Bundles を使用する。

---

### 3-2. ブランチ戦略とプルリクエストワークフロー

推奨されるブランチ戦略:

```
[ローカル開発 / 開発ワークスペース]
    ↓ feature ブランチ作成
[フィーチャーブランチ]
    ↓ プルリクエスト → レビュー → マージ
[main / release ブランチ]
    ↓ CI/CD による自動デプロイ
[ステージングワークスペース] → テスト通過後 → [本番ワークスペース]
```

**ベストプラクティス:**
- 変更は必ずフィーチャーブランチで開発し、main への直接プッシュは避ける
- ステージング環境でのテストが通過した場合のみ本番へデプロイ
- 競合（コンフリクト）は Git の標準的な解決フローで対処

---

### 3-3. テスト戦略

| テスト種別 | 範囲 | 目的 |
|----------|------|------|
| **ユニットテスト** | 個別の関数・変換ロジック | ロジックの正確性を検証 |
| **統合テスト** | 複数コンポーネントの組み合わせ | コンポーネント間の連携を検証 |
| **E2E テスト（エンドツーエンド）** | パイプライン全体 | 実環境に近い条件で全体フローを検証 |
| **UAT（ユーザー受け入れテスト）** | ビジネス要件との適合 | ビジネスロジックの正確性を確認 |

Python の `pytest` フレームワークが Databricks でのユニットテストによく使われる。

---

### 3-4. Databricks Asset Bundles（DABs）

DABs は、Databricks のリソース（ジョブ、パイプライン、クラスター設定など）を **YAML ファイルとして定義**し、複数環境に一貫してデプロイするための Infrastructure as Code（IaC）フレームワーク。

**DABs が解決する課題:**
- Git フォルダーでは管理できないジョブ・パイプライン設定をコード化
- 開発・ステージング・本番の環境差分（ワークスペース URL、クラスターサイズなど）を変数で管理
- CI/CD パイプライン（GitHub Actions、Azure Pipelines）からの自動デプロイを実現

**バンドルの構成:**

```yaml
# databricks.yml の基本構造
bundle:
  name: my-data-pipeline

targets:
  dev:
    workspace:
      host: https://dev.azuredatabricks.net
  prod:
    workspace:
      host: https://prod.azuredatabricks.net

resources:
  jobs:
    my_job:
      name: "My Data Pipeline Job"
      tasks:
        - task_key: ingest
          notebook_task:
            notebook_path: ./notebooks/ingest
```

**主要ファイル構成:**

| ファイル | 役割 |
|---------|------|
| `databricks.yml` | バンドルのルート設定ファイル（ターゲット環境・リソース定義） |
| `resources/*.yml` | ジョブ・パイプライン・クラスターの個別設定 |
| `src/` または `notebooks/` | ノートブック・Python ソースコード |

---

### 3-5. Databricks CLI によるデプロイ

```bash
# バンドルの検証
databricks bundle validate

# 開発環境へのデプロイ
databricks bundle deploy --target dev

# 本番環境へのデプロイ
databricks bundle deploy --target prod

# ジョブの実行
databricks bundle run --target prod my_job
```

**DABs vs Git フォルダーの比較（試験重要）:**

| 観点 | Git フォルダー | Databricks Asset Bundles |
|------|--------------|--------------------------|
| **管理対象** | コードファイルのみ | コード + ジョブ/パイプライン設定すべて |
| **環境差分管理** | 不可 | YAML 変数で環境ごとに上書き可能 |
| **CI/CD との統合** | 限定的 | CLI コマンドで完全自動化が可能 |
| **推奨用途** | 軽量な個人開発・実験 | 本番グレードのチーム開発・マルチ環境デプロイ |

## モジュール 4: ワークロードの監視・トラブルシューティング・最適化

### 学習目標

- メトリクス・自動終了・予算を使用してクラスター消費を監視・管理できる
- Jobs UI と Repair Run 機能を使用して失敗した Lakeflow Jobs をトラブルシューティングできる
- Spark UI とコンピュートメトリクスを使用して Spark ジョブの障害とリソースボトルネックを診断できる
- キャッシュ・データスキュー・メモリスピル・シャッフルのパフォーマンス問題を調査・解決できる
- Azure Databricks から Azure Log Analytics へのログストリーミングを実装できる

---

### 4-1. クラスター消費の監視と管理

| 監視項目 | 確認場所 | 対処方法 |
|---------|---------|---------|
| **DBU 消費量** | Azure Cost Management / Databricks 管理コンソール | 予算アラートの設定、クラスターポリシーの適用 |
| **クラスター使用率** | クラスターメトリクスページ | 自動スケーリングの設定 |
| **アイドル状態** | クラスター設定 | 自動終了（Auto Termination）で未使用クラスターを停止 |

**自動終了（Auto Termination）の重要性**: 使用されていないインタラクティブクラスターは、自動終了を設定しないとコストが発生し続ける。ジョブクラスター（ジョブ専用クラスター）はジョブ完了時に自動的に終了するため、コスト最適化に有効。

---

### 4-2. Lakeflow Jobs のトラブルシューティング

**失敗したジョブの診断フロー:**

```
1. Jobs UI でジョブ実行の詳細を確認
2. 失敗したタスクのログ・エラーメッセージを確認
3. ドライバーログ / Spark ログでスタックトレースを確認
4. 問題を修正後、Repair Run で成功済みタスクをスキップして再実行
```

**Repair Run（実行の修復）**: 失敗したジョブ実行を、成功済みタスクを再実行することなく失敗タスクのみ再実行する機能。長時間のバッチジョブで特に効果的。

---

### 4-3. Spark UI によるジョブ診断

Spark UI は、Spark アプリケーションのリアルタイムおよび履歴情報を提供する Web ダッシュボード。

**Spark UI の主要タブ:**

| タブ | 確認できる情報 |
|-----|--------------|
| **Jobs** | Spark ジョブの一覧・実行時間・成否 |
| **Stages** | ステージごとの処理時間・タスク数・データ量 |
| **Tasks** | 個別タスクのメトリクス（入出力、シャッフル量など） |
| **SQL / DataFrame** | クエリプランと実行時間（Photon 使用時も確認可） |
| **Storage** | キャッシュされた RDD / DataFrame の状態 |
| **Environment** | Spark 設定・JVM 情報 |

**DAG（有向非巡回グラフ）の活用**: ステージページの DAG ビジュアライゼーションで、RDD / DataFrame の変換フローと依存関係を視覚的に確認できる。ボトルネックとなるステージを特定するために活用する。

---

### 4-4. パフォーマンス問題の診断と解決

#### データスキュー（Data Skew）

**症状**: 一部のタスクが他のタスクより著しく遅い（Straggler タスク）。

| 原因 | 解決策 |
|------|--------|
| 特定のキー値にデータが集中 | Adaptive Query Execution（AQE）の活用（デフォルト有効。256MB 超かつ平均の 5 倍超のパーティションを自動分割） |
| JOIN キーに NULL や固定値が多い | スキューヒントの明示、データの前処理でスキューを軽減 |

#### メモリスピル（Memory Spill）

**症状**: タスクが遅い、ディスク I/O が多い。

| 原因 | 解決策 |
|------|--------|
| シャッフル時にメモリ不足 | クラスターメモリの増加、パーティション数の調整（`spark.sql.shuffle.partitions`） |
| 大量データの集計 | ブロードキャストジョイン（小さいテーブルをブロードキャスト）の活用 |

#### シャッフル（Shuffle）

シャッフルはネットワーク経由でのデータ再分配であり、最もコストの高い処理の一つ。

| 最適化手法 | 説明 |
|----------|------|
| **パーティション数の調整** | `spark.sql.shuffle.partitions`（デフォルト 200）を実データ量に合わせて調整 |
| **ブロードキャストジョイン** | 小さいテーブル（デフォルト 10MB 以下）をブロードキャストして JOIN のシャッフルを回避 |
| **AQE（Adaptive Query Execution）** | 実行時の統計情報に基づいてクエリプランを動的に最適化 |

#### キャッシュ（Caching）

| キャッシュ方式 | 特徴 | ユースケース |
|-------------|------|------------|
| `df.cache()` / `df.persist()` | メモリ（またはメモリ+ディスク）に DataFrame をキャッシュ | 複数回参照される中間 DataFrame |
| **Delta キャッシュ** | ディスクベース、SSD へのデータキャッシュ | ホットデータへの高速アクセス |
| **Spark UI の Storage タブ** | キャッシュ状態を確認 | 不要なキャッシュのクリアや確認に活用 |

---

### 4-5. Delta テーブルの最適化（OPTIMIZE / VACUUM）

| コマンド | 説明 | 推奨実行タイミング |
|---------|------|----------------|
| `OPTIMIZE <テーブル名>` | 小さなファイルを 128MB 単位にコンパクション | 毎日のバッチジョブ後（夜間実行推奨） |
| `OPTIMIZE <テーブル名> ZORDER BY (<列名>)` | Z-ORDER クラスタリングでデータの局所性を向上（特定列でのフィルタリング高速化） | フィルタリング頻度の高い列に対して実行 |
| `VACUUM <テーブル名>` | 保持期間（デフォルト 7 日）を超えた古いファイルを削除 | OPTIMIZE 後や定期的なメンテナンス時 |
| `VACUUM <テーブル名> RETAIN <N> HOURS` | 指定時間分の履歴を残して削除 | タイムトラベル要件に応じて調整 |

**液体クラスタリング（Liquid Clustering）**: Z-ORDER の後継として登場した動的クラスタリング手法。静的パーティションや Z-ORDER と異なり、クラスタリングキーを後から変更でき、OPTIMIZE を実行するたびに自動的にデータを再配置する。将来の最適化選択肢として重要。

### 4-6. Azure Log Analytics へのログストリーミング

Azure Databricks のログを Azure Monitor の Log Analytics ワークスペースに集約することで、一元的な監視と長期的なログ保持が可能になる。

**ログストリーミングの設定手順（概要）:**

```
1. Azure Log Analytics ワークスペースを作成
2. Databricks クラスター設定で Log Analytics エージェントを有効化
   - ワークスペース ID と主キーを設定
3. クラスタースコープの init スクリプトで Log Analytics エージェントをインストール
4. Azure Monitor でカスタムクエリ（KQL）を使用してログを分析
```

**Azure Monitor との統合メリット:**

| 機能 | 説明 |
|------|------|
| **一元管理** | 複数ワークスペースのログを単一の Log Analytics に集約 |
| **長期保持** | Databricks デフォルトより長い保持期間の設定が可能 |
| **アラート設定** | ログの特定パターンに基づくアラート（Azure Monitor アラートルール） |
| **KQL クエリ** | Kusto Query Language で柔軟なログ分析が可能 |

| モジュール | 最重要コンセプト |

| **M3: 開発ライフサイクル** | DABs によるマルチ環境デプロイ / Git フォルダーとの違い |
| **M4: 監視・最適化** | Spark UI による診断 / スキュー・スピル・シャッフルの解決策 / OPTIMIZE・VACUUM |

### 試験頻出の比較ポイント

| 比較テーマ | 選択基準 |
|----------|---------|
| **ノートブック vs Lakeflow SDP** | カスタムロジックが複雑 → ノートブック / CDC・ストリーミング・データ品質 → Lakeflow SDP |
| **Git フォルダー vs DABs** | 個人・コード管理のみ → Git フォルダー / チーム・マルチ環境・設定管理も必要 → DABs |
| **スケジュール vs イベントトリガー** | 定期バッチ → Cron スケジュール / データ到着時に即実行 → ファイル到着 / テーブル更新トリガー |
| **Z-ORDER vs 液体クラスタリング** | 既存テーブルの最適化 → Z-ORDER / 新規テーブルで柔軟なクラスタリング → 液体クラスタリング |
| **ジョブクラスター vs オールパーパスクラスター** | 本番バッチ実行（コスト最適） → ジョブクラスター / 開発・対話的作業 → オールパーパスクラスター |

### Lakeflow Spark Declarative Pipelines（旧 DLT）の進化

Delta Live Tables（DLT）は 2025 年に Lakeflow プラットフォームの一部として **Lakeflow Declarative Pipelines** に統合・改称された。宣言型変換レイヤーとして Lakeflow Jobs（オーケストレーション）と連携し、エンドツーエンドのデータパイプライン基盤を構成する。
